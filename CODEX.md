# Spend Sense Codex Guide

Last updated: 2026-05-27

This file is the working guide for Codex sessions in this repository. It captures the current architecture, expected commands, and the backend direction agreed for Spend Sense.

## Product Context

Spend Sense is a privacy-first personal finance app. The frontend currently stores budgets, expenses, savings goals, settings, family members, challenges, and badges in browser localStorage through Zustand.

The backend has been scaffolded with FastAPI and now includes the authentication foundation:

- `backend/app/main.py` creates the FastAPI app, CORS middleware, root route, and API v1 router.
- `backend/app/api/v1/routes/health.py` exposes `GET /api/v1/health`.
- `backend/app/api/v1/routes/auth.py` exposes register, login, refresh, current-user, and logout endpoints.
- `backend/app/db` contains SQLAlchemy session/base setup.
- `backend/app/models/user.py` contains the auth SQLAlchemy models.
- Alembic has an auth migration for `users` and `refresh_tokens`.

Do not implement business features beyond authentication unless the user explicitly asks for that phase to begin.

## Common Commands

Frontend:

```powershell
npm run dev
npm run build
npm test
npm run lint
```

Backend, from `backend/`:

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
uvicorn app.main:app --reload
alembic upgrade head
```

If running backend commands from the repository root, set `PYTHONPATH=backend` or run commands with `backend` as the working directory.

## Backend Direction

The backend should move in phases:

1. Authentication foundation.
2. Profile, preferences, and local-store import.
3. Budget, category, and expense persistence.
4. Savings goals, contributions, analytics, and reports.
5. Family wallet, shared expenses, and settlements.
6. Gamification and insights.
7. Hardening, observability, and deployment.

See [docs/backend-roadmap.md](docs/backend-roadmap.md) for the detailed plan.

## Data Rules

- Use PostgreSQL as the source of truth once persistence begins.
- Use `uuid` primary keys generated by PostgreSQL.
- Use `numeric(14,2)` for money. Do not use floating point types for financial values.
- Store dates as `date`, timestamps as `timestamptz`.
- API month values should use `YYYY-MM`; database month values should be first-day-of-month `date`.
- Preserve the existing frontend domain concepts while allowing server-grade normalization.
- Treat localStorage migration as a first-class requirement.

The finalized schema specification lives in [docs/postgresql-schema.md](docs/postgresql-schema.md).

## API Rules

- Version all backend endpoints under `/api/v1`.
- Keep unauthenticated endpoints limited to root, health, docs, and auth entry points.
- Use bearer access tokens and refresh tokens for authenticated sessions.
- Return ISO 8601 timestamps and `YYYY-MM-DD` dates.
- Use stable error shapes so the frontend can handle validation and auth errors consistently.

The API contract specification lives in [docs/api-contracts.md](docs/api-contracts.md).

## Implementation Guardrails

- Keep backend feature work behind explicit user requests.
- Prefer small, reviewable changes that align with the existing FastAPI structure.
- Add Pydantic schemas before exposing route handlers.
- Add SQLAlchemy models and Alembic migrations together.
- Add repository/service tests for persistence logic before wiring broad API behavior.
- Do not remove or rewrite the current local-first frontend state model until a migration path exists.
- Do not store raw refresh tokens, passwords, or receipt file contents in plain text.

## Verification Expectations

For frontend-only changes:

- Run `npm run build`.
- Run focused tests or `npm test` when behavior changes.

For backend changes:

- Run `python -m compileall app` from `backend/`.
- Run backend tests once they exist.
- Run Alembic migration generation/upgrade checks when schema changes.

For full-stack contract changes:

- Update [docs/api-contracts.md](docs/api-contracts.md).
- Keep TypeScript types and Pydantic schemas aligned.
- Note any frontend migration requirements in the roadmap or PR summary.
